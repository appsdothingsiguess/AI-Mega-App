I'm evaluating Cactus Compute's "Needle" (github.com/cactus-compute/needle), a
26M-parameter distilled model for single-shot on-device tool/function calling.
On my own real tool registry (6 tools, mixed single- and multi-argument
schemas: web_search, fetch_url, file_read, file_grep, run_code, memory_save),
the untuned base checkpoint scored only 25.2% call-F1 / 25% exact-match on a
60-example held-out test set, and got 0/10 correct on 3 of the 6 tools
(file_grep, file_read, web_search) -- much worse than the demo/toy examples
in Needle's own README/playground suggest.

I ran a local finetune rehearsal (250 synthetic examples, ~41/tool, 3 epochs)
and only reached 42% call-F1 -- Needle's own docs recommend >=120 examples
per tool, and I only had a third of that, so this isn't a real result yet.

Questions:

   on a *custom* tool schema (not the demo toolset)? Blog posts, GitHub
   issues, Discord/Reddit threads, HN comments, X/Twitter threads -- anything
   beyond Cactus's own marketing material.
2. What's the actual recommended/working finetuning recipe people have used

   training data (my finetune barely moved base accuracy after only ~3 total
   gradient steps at batch=64/130 examples -- is that a known failure mode?).
3. Are there known accuracy gaps specific to certain tool-schema shapes

   with no x86 fallback -- is there a known workaround, alternate runtime, or
   community port that avoids that?
5. Is there consensus on when Needle is/isn't a good fit vs. just using a
   small (1-4B) general instruct model prompted directly for single-shot tool
